The Problem

Smart but Unsupervised

Claude knows Salesforce. It can write Apex, design Flows, scaffold LWC components. But training data has three problems that compound in production.

⚠️

Outdated

Training data has a cutoff. New platform features, API changes, and deprecated patterns aren't reflected. Claude might generate code using @future methods when Queueable with Finalizer is the current best practice.

🎯

Approximate

Claude knows Apex syntax but doesn't enforce governor limits, sharing model defaults, or CRUD/FLS checks unless prompted. It generates code that compiles but may not survive a security review.

❌

Missing

Some things simply don't exist in training data. Agent Script — Salesforce's language for authoring Agentforce agents — launched in 2025 with zero training data in any AI model. Claude will pattern-match from Python and produce plausible but broken code.

The real cost isn't wrong code — it's confident wrong code. Claude will generate Agent Script that looks syntactically reasonable, uses 4-space indentation, defines subagents and actions. It compiles in your head. It fails on the platform. And debugging "why does this .agent file not work" without knowing the actual language spec is a dead end.

I built an intelligence platform with 7 Agentforce topics and 63 agent actions. Each action needed to be wired to an Apex invocable method through Agent Builder — manually scanning classes, mapping I/O parameters, debugging routing. It took weeks across multiple sessions.

The isLocal Problem — Solved

Agent Builder defaults actions to isLocal=true — mocked, not connected to real data. You discover this at publish time after manual testing. The skill pipeline catches it earlier: preview first with mocked actions to test routing, then --use-live-actions to verify real data flows. The publish checkpoint requires live-action preview to pass. No more silent isLocal failures in production.

Proof Point

The Meridian Agent Build

A full 7-topic, 68-action Agentforce agent — built from scratch, published to a live org, entirely through Agent Script CLI. While the owner did yard work.

It didn't work on the first try. It took 6 iterations. That's the real story.

The Iteration Chain

Try 1
FAIL — Split into multiple files. Compiler: "Missing config/system/start_agent." Learning: Agent Script is one file per agent.
Try 2
FAIL — Actions referenced by name only. Server: "'action_name' is not defined." Learning: Actions need target definitions.
Try 3
FAIL — Targets inline in reasoning block. Server: "unexpected ://" Learning: Topic has two levels — actions: defines, reasoning.actions: references.
Try 4
FAIL — Top-level actions block. Server still rejected. Learning: actions: is per-topic, sibling of reasoning:.
Try 5
CLOSE — Correct structure! But output schema mismatches (Apex names vs Flow names). Learning: Flows expose different I/O than Apex classes.
Try 6
SUCCESS — Parsed flow XML for ground-truth schemas. Published. Live in org.

What Unblocked It

  • › Salesforce Docs MCP — searched live docs mid-session, found the canonical two-level pattern that no training data had
  • › sf agent validate — server compiler gave exact line numbers + "did you mean?" suggestions
  • › Flow XML parsing — extracted actual I/O schemas from 38 flow files programmatically
  • › Tooling API — deleted orphaned action, assigned permissions, queried metadata — all without opening a browser

Skills + Tools Used

  • › /sf-ai-agentscript — DSL reference, validator, architecture patterns
  • › /developing-agentforce — metadata lifecycle, safety review
  • › salesforce-docs MCP — live doc search (the breakthrough)
  • › sf agent validate/publish — server-side compilation + deploy
  • › sf data create record — programmatic permission assignment
  • › sf api request rest — Tooling API for orphan cleanup
